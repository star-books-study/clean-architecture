# Chapter 01. 계층형 아키텍처의 문제는 무엇일까?

## 계층형 아키텍처는 데이터베이스 주도 설계를 유도한다
- 전통적 계층형 아키텍처의 토대 : 데이터베이스
- 보통 데이터베이스 구조를 먼저 생각 -> 이를 바탕으로 도메인 로직을 구현함
  - 이것이 의존성 방향에 따라 자연스럽게 구현한 것이기 때문
- 다만, **비즈니스 관점에서는 도메인 로직을 먼저 만들어야 로직을 제대로 이해한 것인지 확인할 수 있다.**
  - 상태가 아니라, `행동`을 중심으로 모델링한다. 
  - 행동이 상태를 바꾸는 주체이기 때문에 행동이 비즈니스를 이끌어간다
- ORM 프레임워크를 계층형 아키텍처와 결합하여 쓰는 순간, 비즈니스 규칙을 영속성 관점과 섞고 싶어진다.
  - ORM 에 의해 관리되는 엔티티는 일반적으로 영속성 계층에 두게 되며, 도메인 계층은 이 엔티티에 접근하여 사용할 수 있게 된다
  - 이 시점에 **영속성 계층 <> 도메인 계층 간 강결합이 발생된다**

## 지름길을 택하기 쉬워진다
- **영속성 계층이 여러 컴포넌트들이 섞이면서 더욱 비대해진다**
  - 어떤 계층에도 속하기 애매하게 여겨지는 헬퍼, 유틸리티 컴포넌트들이 영속성 계층 레벨로 내려오기 쉽다

## 테스트하기 어려워진다
- 한번 계층을 건너뛰는 케이스가 발생한다면, **애플리케이션 전반에 걸쳐 책임이 섞이고 핵심 도메인 로직이 퍼져나갈 확률이 높다**

## 유스케이스를 숨긴다
- 계층형 아키텍처는 **도메인 서비스의 '너비' 에 대한 규칙을 강제하지 않는다**
- 따라서 도메인 로직이 여러 계층에 걸쳐 흩어져있기 쉬우므로, **하나의 서비스가 여러 개의 유스케이스를 담당하는 아주 넓은 서비스가 만들어지기도 한다**

## 동시 작업이 어려워진다
- 코드에 여러 유스케이스를 담당하는 너비가 넓은 서비스가 존재할 경우, **서로 다른 기능을 같은 서비스에 대해 동시 편집하는 상황 발생** -> 병합 충돌 가능성
